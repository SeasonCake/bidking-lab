# 工程理解札记

本文件记录项目推进中值得保留的工程判断。它不是进度日志，而是后续设计、重构和解释项目核心思路时可复用的原则。

## Canonical Input 灰度切换

当系统要从旧输入路径切到新输入路径时，不应直接替换调用点。更稳的做法是：

1. **双路径并存**：旧路径继续服务用户，新路径先作为 shadow state 收集同一批输入。
2. **统一输出合同**：两条路径都必须转换成同一个领域对象，这里是 `SessionObs`。
3. **稳定字段展开**：不要直接比较 dataclass 对象，而是展开为稳定字段路径，例如 `bucket.4.avg_cells`、`bucket.6.total_cells`。
4. **差异可视化**：灰度阶段把 legacy/live 的差异展示给用户和开发者，而不是只显示“结果不同”。
5. **默认可回退**：新路径打开后若上下文不匹配，应自动回退旧路径；只有真实场景差异长期稳定为 0 后，再考虑切默认。

这套流程的价值是把“切换输入源”从一次性风险操作，变成可观测、可回滚、可用真实截图逐步验证的过程。

## 诊断层不应改变推理

来源摘要、blocked update、live/legacy diff 都属于诊断层。它们可以暴露状态、解释覆盖规则、帮助判断是否能切默认，但不应该偷偷改变 MC、joint 或分析估算的输入。

如果诊断面板本身会改变推理结果，就会让问题更难定位。因此诊断层只读状态，推理入口必须显式选择输入源。

## 诊断 UI 依赖要轻

诊断面板应该尽量使用基础 UI 能力。像 `st.dataframe()` 这类组件会依赖 Arrow / pyarrow / numpy 的二进制组合；一旦本机环境不匹配，诊断层反而会阻塞主应用启动。

小表格、来源摘要、diff 行这类信息优先使用 Markdown 表格即可。它牺牲少量交互能力，但启动路径更稳，也符合“诊断层不影响推理”的边界。

## 事件驱动刷新优先于持续轮询

实时推理不应该因为“时间过去了”就重算。更合理的触发源是有语义的游戏事件，例如轮次变化、道具读数出现、公开信息变化、手填/OCR 更新、换图或新局。

这意味着 UI 状态机可以很小：

- `dirty`：live 观测变化，旧结果可能过期。
- `running`：后台推理正在消费当前输入。
- `ready`：已有结果，且 live 输入没有新的语义变化。
- `error`：推理失败，需要用户看到失败而不是误读旧结果。

先显示状态，再自动调度重算。这样能先验证“什么时候应该变 dirty”，再决定“什么时候启动后台任务”，避免实时接入后刷新过于频繁。

自动调度还需要一个“武装”条件：只有成功产出过一次推理结果，并且该 live 状态被 `mark_ready()` 标为新鲜后，后续 dirty 才能触发自动重算。这样初次打开 app 不会因为 shadow state 初始化而预热 MC，实时刷新只服务于“已有结果变旧”的场景。

## 截图回归的可靠形态

OCR 引擎依赖外部模型和本机环境，适合作为 slow smoke，不适合作为每次必跑的核心回归。更稳定的做法是：

- OCR/解析层保留少量 slow 图片 smoke。
- 输入适配层使用“截图派生 obs”作为普通单测。
- 缺失仓库总格时使用明确默认值，例如当前约定的 `123`，避免测试依赖 UI 临时状态。

这样既能保留真实图片语义，又不会让核心回归被 OCR 安装状态拖垮。

## 估计值不要伪装成精确值

伊森 R1-R4 的仓库底部轮廓是有价值的先验，但不是容量真值。把它直接写成
`warehouse_total_cells` 会让 joint / MC 把“估错的底部站位”当作硬上限，过早剪掉真实组合。

更稳的建模方式是分两类字段：

- `warehouse_total_cells`：OCR、R5 技能、packet 精确状态给出的严格事实。
- `warehouse_total_cells_approx` + `warehouse_total_cells_tolerance`：R1-R4 手填估计，按“估计值+容差”做剪枝上限。

小仓库的容差要比大仓库更克制，否则几个格子的误差会变成很大的相对差异。第五回合若得到完整轮廓或物品列表，再写入精确总格和 `total_item_count`。
