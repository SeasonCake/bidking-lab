## 近期实战问题与修复结论

1. Ahmad R1 总件数未显示

- 原因：`100204` 结果在 `field7`，旧 `_parse_skill_reveal()` 漏字段；
- 修复：skill reveal 整数字段候选加入 `field7`；
- 验证：raw sort 1 能解析 `100204 result=39 field=7`，live batch 产生 `session.total_item_count=39`。

2. 良品存量未进入蓝件

- 原因：普通 action result 也有 `field7`，旧 `_parse_action_result()` 漏字段；
- 修复：action result 整数字段候选加入 `field7`；
- 验证：`100117 result=13 field7` 进入 `fixed_counts.q3=13`。

3. 手填后不再跟随实时抓包

- 原因：旧手填路径把 UI 切成孤立 manual snapshot；
- 修复：manual overlay 合并到最新 live snapshot；
- 预期：手填启用后仍保留 live hero/map/round/session/minimap，后续抓包继续刷新。

4. Victor `100209` 语义错误

- 原因：旧记录按 `q4+q5`，实战文本实际是 `q4+q5+q6`；
- 修复：live parser、hero skill metadata、ref engine、manual UI、测试统一改为 `q4q5q6`；
- 风险：旧文档/旧样本中可能仍出现 `q4q5`，只能作历史兼容。

5. 金均格 0 未进入推理

- 原因：部分结果包没有显式 result 行，且 parser 旧逻辑把 0 当缺失；
- 修复：支持 varint 0；必要时用带标记的 `inferred_zero`；
- 风险：如果网络丢了非零结果包，`inferred_zero` 仍可能误判，所以 UI/notes 必须显示来源。

6. 均格乘件数产生小数格

- 原因：旧逻辑可能直接用 `count * avg` 当总格；
- 修复：统一通过 `_avg_grid_options()` 选合法整数格；
- 验证：`avg=1.8` 下只有 `5件 -> 9格` 合法。

7. Sparse evidence 卡 UI

- 观察：`总件38 + 金均格9.0` 这类弱约束会让组合枚举耗时 15 秒以上；
- 结论：不能靠简单降低 `max_combos` 解决，低 cap 会因枚举顺序偏置导致价格失真；
- 后续方案：若仍遇到卡顿，应把 summary 放后台线程/进程或加可解释的 early bound，而不是砍组合数。
