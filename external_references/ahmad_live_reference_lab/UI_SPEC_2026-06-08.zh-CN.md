# Ahmed Live UI Spec

日期：2026-06-08

## UI 目标

旧 UI 的问题是体积大、信息散、推理依据不透明。Ahmed 实验 UI 应该小而清晰，把“用户每轮出价前真正需要看的信息”放在第一屏。

实现方向采用与当前 `scripts/run_live_overlay.py` 一致的 Tkinter 桌面悬浮窗，而不是 HTML 页面。HTML/HTTP 原型只作为早期调试 API，不作为最终 UI。

## 三层 UI

### Level 1：实战 compact overlay

用于局内悬浮/小窗，默认显示：

- 英雄、地图、轮次、更新时间；
- 推荐价三档：保守、参考、激进；
- 红品数量：exact 或 P10/P50/P90；
- 总值区间：P50/P90 或参考/上限；
- 主要风险标签：`低估风险`、`活动图`、`缺表 fallback`、`证据不足`、`过冲警戒`；
- 一条短 rationale：例如 `R2 金均格 + 总件数约束，q6 未确认，使用 villa prior`。

风格要求：

- 保持 bidking-lab 当前 overlay 的深色、紧凑、卡片式视觉；
- 面积应小于当前主 overlay，可常驻游戏边缘；
- 三档价格和红品件数固定在第一屏；
- 不使用浏览器窗口作为默认入口。

不在 Level 1 显示：

- 大段样本统计；
- raw packet；
- OCR 面板；
- 全部 shadow/audit 字段。

### Level 2：Ahmed calculator panel

用于用户想确认为什么价格变化：

| 区块 | 内容 |
|---|---|
| 输入证据 | R1-R5 Ahmed 技能、公开信息、道具扫描、来源优先级 |
| 品质表 | q1/q3/q4/q5/q6 的 count、cells、avg、value band |
| 可达性 | 均格是否唯一/窄/宽/冲突 |
| 推荐解释 | 三档价格来自哪些组合、是否被 safety/guard 限制 |
| 诊断对照 | 仅在需要排查时显示外援参考值 vs v3 practical/live 值，不进入主报价 |

### Level 3：高级/OCR/诊断

默认折叠，只给调试使用：

- OCR 文本输入/截图入口，作为 fallback；
- raw latest_snapshot/model_eval 字段；
- q6 posterior 分布；
- prior drift、activity mapping、CSE/SCP/shadow 标签；
- archive replay 对照。

## 交互细节

- 读数来源用短标签：`packet`、`public`、`scan`、`manual`、`ocr`、`prior`。
- exact 事实使用实线强调；derived/prior 使用淡色。
- 红品件数和估值区间要固定位置，避免每次刷新跳动。
- 价格变化必须展示“触发原因”：新轮次、技能结果、公开信息、扫描、结算或手填。
- 如果使用活动图 fallback，地图名旁边显示 `按旧沉船估` 或 `按 villa family 估`。
- 如果缓存本机 player_id 识别英雄，UI 不需要显示技术细节；只在诊断层显示。
- 如果缺 Ahmed `100204` 总件数，Level 1 应显示 `缺总件数` / `等待外援输入`，不能把均格-only 或未完成 v3 结果伪装成外援 live-ready 推荐。
- 如果 ref_v0 与 v3 分歧超过 10w，只在诊断/交接记录中保留，不在 Ahmad compact UI 主界面干扰实战判断。

## 第一版布局草案

```text
Ahmed · 别墅 2406 · R3      packet 14:57:12

保守  43.5w       参考  52.8w       激进  64.4w
红品  2 / 3 / 3 件      总值 P90 64.4w

风险：活动图 · 证据不足 · 等待总件
依据：总件数 + 金均格 + 紫均格，未见红品 exact

[展开 Ahmed 计算器] [高级诊断]
```

## 显示边界

- UI 可以显示外援参考建议，但必须标注 `external reference` 或 `Ahmed ref`；
- 如果 v3 practical 与 Ahmed ref 分歧大，默认仍只显示 Ahmed ref；差异和原因留在诊断层或文档中；
- 正式出价接线由主线决定，本实验 UI 只给参考。
