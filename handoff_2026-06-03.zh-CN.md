### WinDivert monitor（`scripts/run_windivert_live_monitor.py`）

- `capture_source_status.json` 含 `accepted_frames`、`ignored_reasons`、`active_session_id`。

### Monitor 写盘（`scripts/run_fatbeans_webhook_monitor.py`）

- **Fast bootstrap 保护**：`fast` 且 `rows<3` 且推理 `phase=settled` 时 **跳过** 写入 `latest_snapshot.json`，避免新局首帧携带「上一局结算快照」污染 UI（日志：`[wait] skip fast bootstrap settled snapshot rows=1`）。

### Overlay（`scripts/run_live_overlay.py`）

- 读取 snapshot 时合并 `capture_source_status.json`（`_capture_source_status`）。
- 若快照为上一局 `settled`，且抓包 `active_session_id` 已变（或快照无 `session_id`），显示 **「新局监听中」**，不继续钉死旧结算数字。
- 测试：`tests/test_live_overlay.py`（session-ahead / stale settlement）。

## 用户最新实测问题（2026-06-03，优先级排序）

### P0-1 · 英雄错识：艾莎 → 伊森

**现象**：开局选艾莎，UI/快照显示 `ethan`。

**根因（高置信）**：`src/bidking_lab/live/fatbeans.py` 中 `_hero_mode_from_state()` 在状态帧里按顺序取 **第一个** 带 `hero_id` 的 `skill_reveal` 或 **`bids` 列表中的玩家**；大厅/开局状态帧常含 **其他玩家** 的 `hero_id=208`（伊森），早于或覆盖本地艾莎。

```904:917:src/bidking_lab/live/fatbeans.py
def _hero_mode_from_state(state: FatbeansStateEvent) -> str | None:
    for reveal in state.skill_reveals:
        ...
    for bid in state.bids:
        ...
        hero = _HERO_MODE_BY_ID.get(bid.hero_id)
```

**待办**：用 `0x0021` 会话起点、本地 `player_id`、或「仅更新己方 hero」规则；勿用全桌第一个 bid 的 hero_id。

### P0-2 · 局末无结算视图，直接「新局监听中」

**现象**：第一局结束应显示结算 truth，实际立刻进入「新局监听中」；第二局包仍难解析。

**根因（组合）**：

1. **Overlay 会话领先逻辑过早**：`active_session_id` 在新局 TCP/状态帧出现时即更新，而 `latest_snapshot.json` 仍为上一局 `settled` → `_capture_session_ahead_of_snapshot()` 为真 → **立即** 走「新局监听中」，**抢在** 用户查看结算之前（`SETTLED_RESULT_RETAIN_SECONDS=60` 分支排在后面）。
2. **Fast bootstrap 跳过**：局末若只有 1–2 行就触发 fast 且被判 `settled`，可能 **不写** 最终快照，overlay 仍显示更早的结算或待机。
3. **第二局抓包**：`accepted_frames` 仍偏低（实测曾 **8**）、`rev_not_game_frame` 高、`send_wrong_session`；flow 切换清缓冲后常 `frames=1` 就推理。

**待办**：

- Overlay：结算保留期内 **优先显示 settled truth**，即使 capture session 已变；或要求 snapshot/capture **session_id 显式不等** 且结算年龄 &gt; N 秒才切换「新局监听」。
- Artifact：在 `ui_contract.context` 写入 `session_id`（当前快照常无此字段，导致「只要 capture 有 session 就认为领先」）。
- 第二局：继续查 frame gate 在 lobby/R1 的 `0x0021`/`0x0025` 到达率；必要时局间 **不** `clear_session` on flow change，或延迟 flow-change reset。

### P1 · 前几轮推理不准（轮次未对齐）

**现象**：R1–R3 出价/后验明显偏低或错图；R4+ 有时 `matched 10/10` 仍与桌价差距大。

**可能原因（需实验验证）**：

1. **Live fast 路径**：`fast_n_trials=10`、`roi_trials=0`，证据少时后验方差大（非仅轮次问题）。
2. **轮次语义**：`monitor._action_round()` 用「包内已完成轮次 +1」作可操作轮；`model_eval` / v2 匹配是否按 **相同 round 标签** 过滤历史表样本，需查 `build_monitor_artifact` → v2 problem 是否带 `round`/`evidence_stage`（`early_1_2` / `mid_3_4` 仅用于密度分，未必限制采样池）。
3. **调参历史**：离线 338 样本 replay 多为 **完整局末** 状态；live 早期轮证据结构不同，直接套表会系统性偏低。

**待办**：

- 用 `scripts/summarize_size_bucket_live.py` / `model_eval.jsonl` 按 `round`/`action_round` 分组看 matched 与 P50。
- 实验：`-FastNTrials 40` 或临时 full 推理对比；评估 v2 是否应按 `action_round` 过滤候选样本或加权。
- 文档：`docs/size_avg_interpretation.zh-CN.md` 与 live 轮次读数对照。

### P1 · 火绒拦截 WinDivert

**现象**：火绒开时 `WinDivertOpen` 失败（`winerror=5`）；关闭后 `diagnose_windivert.py` → `result=OK`。

**已知路径**：`%AppData%\Roaming\Python\Python313\site-packages\pydivert\windivert_dll\WinDivert64.sys`（以 diagnose 输出为准）。

**缓解（无完美替换）**：

| 方案 | 说明 |
|------|------|
| 火绒白名单 | 对上述 `.sys`、`python.exe`、`BidKing.exe` 添加信任 |
| 关闭脆弱驱动 blocklist | `HKLM\...\CI\Config\VulnerableDriverBlocklistEnable=0`（需管理员；diagnose 会提示） |
| **Fatbeans WebHook** | `run_fatbeans_webhook_monitor.py`：游戏内 Hook POST，**不依赖 WinDivert**；需 Fatbeans 配置 |
| 历史 JSON 监视 | 非实时，仅批处理评估 |

**无简单替换**：Npcap/Raw Socket 无法等价替代当前 **进程级游戏帧重组 + 0x0027 直响** 链路。
